Command Line Interface

X-CUBE-AI Expansion Package

r2.1
AI PLATFORM r5.2.0 (Embedded Inference Client API 1.1.0)
Command Line Interface r1.4.0

Introduction

The stm32ai application is a console utility which provides a complete and unified Command Line Interface (CLI) to generate from a pre-trained model, an optimized neural network C-library for STM32 device family. It consists on three main commands: analyze, validate and generate. Each command can be used independently of the other with the same set of common options (model files, compression factor, output directory…) and specific options. The quantize command is a specific case to apply a post-training quantization process (refer to [8]).

Synopsis

Neural Network Tools for STM32 v1.4.0 (AI tools v5.2.0)
usage: stm32ai.py [-h] [--version] [--tools-version] [--model FILE]
                  [--verbosity [{0,1,2}]]
                  [--type [keras|tflite|caffe|convnetjs|lasagne|onnx]]
                  [--name STR] [--compression [1|4|8]] [--quantize [FILE]]
                  [--allocate-inputs] [--allocate-outputs] [--workspace DIR]
                  [--output DIR] [--lib DIR] [--series STR] [--split-weights]
                  [--relocatable] [--no-c-files] [--binary] [--address ADDR]
                  [--copy-weights-at ADDR] [--batches INT] [--mode MODE]
                  [--desc DESC] [--valinput FILE [FILE ...]]
                  [--valoutput FILE [FILE ...]] [--full] [--classifier]
                  [--no-check] [--no-exec-model]
                  analyze|generate|validate|quantize

Short description can be displayed with the following command:

$ stm32ai --help

Note — In this article, Netron application (https://github.com/lutzroeder/netron) is used to visualize the original neural network model.

Comparison with the X-CUBE-AI UI plug-in features

Core entry point, the stm32ai application is used as back-end by the X-CUBE-AI UI plug-in (refer to [2]).

In comparison with the X-CUBE-AI UI plug-in, the following high-level features are not supported:

  • extra C-code wrapper to manage multiple models. CLI manages only one model at the time.
  • creation of a whole IDE project including the optimized inference runtime library, AI headers files and the C-files related to the HW settings. CLI can be only used to generate the specialized NN C-files. However, it allows to update an initial IDE project, STM32CubeMX-based or proprietary source tree (see “Update a an ioc-based project” section).
  • the check to know if a model will fit, in term of memory layout in a selected STM32 memory device. CLI reports (see “Analyze command” section) the main system level dimensioning metrics: ROM, RAM, MACC.. (refer to [6] for the definition)
  • for the “Validation process on target”, as a full STM32 project is expected, it must be generated previously through the UI. Note that this project can be updated later (see “Update a an ioc-based project” section). “Validation process on desktop” is fully supported through the CLI without restriction.
  • graphic visualization of the generated c-graph (including the usage of the RAM). CLI provides only a textual representation (table form) of the c-graph including a description of the tensors/operators (see “Analyze command” section).

Command work-flow

For each command, the same preliminary steps are applied. A report (txt file) is systematically created and fully or partially displayed. Additional JSON files (dictionary based) are generated in the workspace to be parsed by the X-CUBE-AI plug-in to retrieve the results. Note that they can be also used by a non-regression environment. The format of these files is out of the scope of this document.

<workspace-directory-path>\<name>_report.json, <name>_c_graph.json
<output-directory-path>\<name>_<cmd_name>_report.txt
  • 'analyze' flow
    • import the model
    • map, render and optimize internally the model
    • log and display a report
  • 'validate' flow
    • import the model
    • map, render and optimize internally the model
    • execute the generated C-model (on desktop or from the STM32 target)
    • execute the original model using original deep learning runtime framework for x86
    • evaluate the metrics
    • log and display a report
  • 'generate' flow
    • import the model
    • map, render and optimize internally the model
    • export the specialized C-files
    • log and display a report

Enable AutoML pipeline for resource-constrained environment

CLI can be integrated in an automatic or manual pipeline, to design a deployable and effective neural network architectures for resource-constrained environment (i.e. with low memory/computational resources and/or critical power consumption budgets). The main loop can be extended with a post-analyzing/validating steps of the pre-trained model candidates to check and to take into account the end-user target constraints thanks to the respective “analyze” and “validate” commands.

  • checking of the budgeted memory (ROM/RAM) can be done in the inner (topology selection/definition) before the time-consuming training process (or re-training process) to pre-constraint the choices of the neural network architecture according the memory budgets.
  • note that the “analyze” and “X86 validate” step can be merged, “analyze” information are also available in the “validate” reports.

Setting the environment

X-CUBE-AI Expansion Package is a complete self-contained application package. No external tools is requested to use the package. %CUBE_FW_DIR% designates the root path where the X-CUBE expansion packages are installed. Default location on Windows:

C:\Users\<user_name>\STM32Cube\Repository

Note — For the generation of a relocatable binary network (refer to [9]), a GNU ARM Embedded tool-chain ('arm-none-eabi-' prefix) should be available in the PATH. In the case where the relocatable binary model is generated through the X-CUBE-AI plug-in, the requested pre-built GNU bare-metal tool-chain for 32-bit ARM processors is automatically downloaded and installed in the installation directory.

Windows® 10

  1. Open a Windows command prompt
  2. Update system PATH variable
set CUBE_FW_DIR=C:\Users\<user_name>\STM32Cube\Repository

set X_CUBE_AI_DIR=%CUBE_FW_DIR%\Packs\STMicroelectronics\X-CUBE-AI\5.2.0
set PATH=%X_CUBE_AI_DIR%\Utilities\windows;%PATH%
  1. [ Alternative solution ] Create an alias with doskey command
doskey stm32ai="%X_CUBE_AI_DIR%\Utilities\windows\stm32ai.exe" $*
  1. Verify the environment
>  stm32ai --version
stm32ai - Neural Network Tools for STM32 v1.4.0 (AI tools v5.2.0)

or

>  stm32ai --tools_version
Neural Network Tools for STM32 v1.4.0 (AI tools v5.2.0)
- Python version   : 3.5.7
- Numpy version    : 1.17.2
- TF version       : 2.3.0
- TF Keras version : 2.4.0
- Caffe version    : 1.0.0
- Lasagne version  : 0.2.dev1
- ONNX version     : 1.6.0
- ONNX RT version  : 1.1.2

Ubuntu® 18.4 and Ubuntu® 16.4 (or derived)

Similar setting of the environment is expected. A native GCC (X86_64) tool-chain should be accessible in the system path.

export X_CUBE_AI_DIR=$CUBE_FW_DIR/Packs/STMicroelectronics/X-CUBE-AI/5.2.0
export PATH=$X_CUBE_AI_DIR/Utilities/linux:$PATH

macOS® (x64)

Similar setting of the environment is expected. A native GCC (X86_64) tool-chain should be accessible in the system path.

export X_CUBE_AI_DIR=$CUBE_FW_DIR/Packs/STMicroelectronics/X-CUBE-AI/5.2.0
export DYLD_LIBRARY_PATH=$X_CUBE_AI_DIR/Utilities/mac
export DYLD_FALLBACK_LIBRARY_PATH=$X_CUBE_AI_DIR/Utilities/mac
export PATH=$X_CUBE_AI_DIR/Utilities/mac:$PATH

Error handling

During the execution of a given command, after the parsing of the arguments if an error is raised, the stm32ai application returns -1 else 0. The description of the error is displayed at the beginning of the summary.

An error message is prefixed by a category and a short description.

category description
LOAD ERROR error during the load/import of the model or the connection with the STM32 board
INVALID MODEL provided network model file is corrupted or cannot be parsed
INVALID OPTIONS specific model parameter is invalid
NOT IMPLEMENTED expected feature is not implemented
TOOLS ERROR INTERNAL ERROR internal error
CLI ERROR specific CLI error
INTERRUPT indicates that the execution of the command has been interrupted by the user (CTRL-C or kill system signal)

Note — There is a specific attention to have explicit and relevant short description of the errors. Unfortunately, this is not always case, additional TIPS and TRICKS can be found in the FAQs [8] article or don’t hesitate to use the ST Community channel/forum or local support.

Common arguments

Following table describes the common arguments for the 'analyze', 'validate' and 'generate' commands. The specific arguments are described in the respective command section.

parameter description
-m/--model indicates the original model file paths (see “DL framework detection” section) - Mandatory
-t/--type indicates the type of original DL framework when it can be not inferred by the extensions of the model files (see “DL framework detection” section) - Optional
-w/--workspace indicates a working/temporary directory for the intermediate/temporary files (default:"./stm32ai_ws/" directory) - Optional
-o/--output indicates the output directory for the generated C-files and report files (default:"./stm32ai_output/"directory) - Optional
-n/--name indicates the C-name (C-string type) of the imported model. Used to prefix the name of specialized NN C-files and the API functions. Also used for the temporary files, this allows to use the same workspace/output directories for different models (default: "network"). - Optional
-c/--compression indicates the expected global factor of compression which will be applied. Supported values: 1|4|8 (default: ‘1’). Refer to [2], “Graph flow and memory layout optimizer” section. Compression can be only performed on the dense-type layer. - Optional
--allocate-inputs if defined, this flag indicates that the “activations” buffer will be also used to handle the input buffers else, default behavior, they should be allocated separately in the user memory space. Depending on the size of the input data, the “activations” buffer may be bigger but overall less than the sum of the activation buffer plus the input buffer. To retrieve the address of the associated input buffers (refer to [5], “IO buffers into activations buffer” section). - Optional
--allocate-outputs if defined, this flag indicates that the “activations” buffer will be also used to handle the outputs buffers, else default behavior, they should be allocated separately in the user memory space. (refer to [5], “IO buffers into activations buffer” section). - Optional
--split-weights if defined, this flag indicates that one c-array is generated by weights/bias data tensor instead to have an unique C-array (“weights” buffer) for the whole (default: disabled), (refer to [5], “Split weights buffer” section) - Optional
-q/--quantize indicates file path of the tensor format configuration file for a Keras model or for the configuration file to perform the Keras post-training quantization process. (refer to [8]) - Optional
-v/--verbosity indicates the level of verbosity (or level of displayed information). Supported values: 0,1,2 (default:1) - Optional

Analyze command

Description

The 'analyze' command is the primary command to import, to parse, to check and to render an uploaded pre-trained model. Detailed report provides the main system metrics to know if the generated code can be deployed on a STM32 device. It includes also rendering information by layer or/and operator (see “C-graph description” section). After completion, the user can be fully confident on the imported model in term of supported layer/operators.

Specific arguments

Only the “Common” arguments are considered.

Examples

  • Analyze a model (simple model file)

    $ stm32ai analyze -m <model_file_path>

  • Analyze a multiple model files (caffe type example)

    $ stm32ai analyze -m ./caffe/lenet.prototxt -m ./caffe/lenet.caffemodel

  • Analyze a 32b float model with compression request

    $ stm32ai analyze -m <model_file_path> -c 8

  • Analyze a model with input tensors placed in activations buffer

    $ stm32ai analyze -m <model_file_path> --allocate-inputs

  • Analyze a Keras post-quantized model (refer to [8])

    $ stm32ai analyze -m <modified_model_file>.h5 -q <quant_file_desc>.json

DL framework detection

Extension of the model files are used to identify the DL framework which should be used to import the model. If the auto-detection is ambiguous, the '--type/-t' option should be used to define the correct framework.

DL framework type (--type/-t) file extension
Keras keras .h5 or .hdf5 and .json or .yml or yaml
TensorFlow lite tflite .tflite
Lasagne lasagne .npz and .py
Caffe caffe .prototxt and .caffemodel
ConvNetJS convnetjs .json
ONNX onnx .onnx

If you try to force a type which is not valid, typical following error is reported

$ stm32ai /c/ai_lab/fxp_demo/demo_asc_fxp/Session_keras_mod_93_Model.h5 --type tflite
...
TOOL ERROR: Invalid extension on input file
    C:\ai_lab\fxp_demo\demo_asc_fxp\Session_keras_mod_93_Model.h5

Initial sub-section recalls the CLI arguments. Note that the full raw command line is saved at the beginning of the generated report file: <output-directory-path>\network_<cmd>_report.txt

field description
model file reports the full-path of the original model files (--model). If multiple files, there is one line by file.
type reports the --type value or inferred DL framework type. For Keras model, the version of the used framework to generate the model is also displayed.
c_name reports the expected C-name for the generated C-model (--name)
compression reports the expected compression factor (--compression)
quantize reports the quantization parameter
workspace dir full-path of the workspace directory (--workspace)
output dir full-path of the output directory (--output)

The second part shows the results of the importing and rendering stages.

field description
model_name designates the name of the provided model. This is generally the name of the model file.
model_hash provides a calculated MD5 signature of the imported model files.
input indicates the name, the item number, the format and the size in bytes of an input tensor. There is one line by input. 'input (total)' field indicates the total size (in bytes) of the inputs.

'input_0 [490 items, 1.91 KiB, ai_float, FLOAT32, (49, 10, 1)]' indicates that input_0 tensor has a size of 490 floating-point items (size in bytes = 490 x 4B = 1.91KiB) with a (49, 10, 1) shape. (refer to [5] “IO tensor” section)
output indicates the name, the format and the size of the output tensor. There is one line by output. output (total) field indicates the total size (in bytes) of the outputs.
param # indicates the total number of parameters of the original model and associated size in bytes.
macc indicates the whole computational complexity of the original model. Value is defined in MACC operations (Multiply-ACCumulated operations) (refer to [6])
weights (ro) indicates the requested size (in bytes) for the generated constant RO parameters (bias and weights tensors). Size is 4-bytes aligned. If the value is different from the original model files, the ratio is also reported. (refer to [6], “Memory-related metrics” section)
activations (rw) indicates the requested size (in bytes) for the working RW memory buffer (also called activations buffer). It is mainly used as internal heap for the activations and temporary results. (refer to [6], “Memory-related metrics” section)
ram (total) indicates the requested total size (in bytes) for the RAM including the input and output buffers.

PINNR/IR graph description

The outlined “graph” section (table form) provides a summary of the topology of the network which is considered before the optimization, render and generation stages. The id column indicates the index of the operator from the original graph. It is generated by the X-CUBE-AI importer. The represented graph is an internal platform independent neural network representation, also called PINNR (or ÌR, internal representation) created during the import. Training only layers are ignored during the conversion. Note that if no input operator is defined, “input” layer is added and generally the layers are un-fused. A complete graphic representation is available through the UI (refer to [2]).

PINNR operator properties.

field description
id indicates the layer/operator index of the original model
layer (type) designates the name of the type of the operator. Name is inferred from the original name. In the case where a layer is un-fused, the new layer is created with the original suffixed with _nl (see next figure with the first layer)
output shape indicates the output shape of the layer. Follow the “HWC” layout or channel last representation: H=height, W=width, C=channel (refer to [5] “IO tensor” section)
param # indicates the number of parameters
connected to designates the name of the incoming layers
macc designates the associated complexity
rom indicates the ROM size (in bytes and 4-bytes aligned)
  • Note that 'rom' size information can be extended with a global indicator about of the C-storage format for the associated weights/bias tensors. See “C-graph description” section.
rom suffix description
nothing indicates that all parameters are 32b float numbers (FP32)
(c) indicates that a part of the parameters is compressed. For a given layer, weights and bias are not necessarily compressed. The report rom size includes also the dictionary.
(q) indicates that a part of the parameters are quantized (fixed-point representation, Qmn or power-of-two scaling integer format). Bias are encoded on 32-bits.
(i) indicates that a part of the parameters are quantized (fixed-point representation, integer format). Bias are encoded on 32-bits.
  • It is possible to have no 'macc' and 'rom' values. This indicates that after rendering and optimizing phases, the referenced layer will be merged with the previous layer (fig. 2 or 3) or as for the reshape operator (fig. 1), the operation is ‘free’.

C-graph description

Additional “Generated C-graph summary” section (also displayed with '-v 2' argument) is included in the report. It summarizes the main computational and associated elements (c-objects) used by the network runtime C-inference engine. It is based on the c-structures generated inside the '<name>.c' file. A complete graphic representation is available through the UI (refer to [2]).

The first part re-calls the main structural elements: c-name, number of c-nodes (implementation of the layer/operator), number of C-array for the data storage of the associated tensors. Input and output name of the I/O network tensors.

Generated C-graph summary
---------------------------------------------------------------------------------------------------
model name         : microspeech_01
c-name             : network
c-node #           : 5
c-array #          : 11
activations size   : 4352
weights size       : 16688
macc               : 336084
inputs             : ['Reshape_1_output_array']
outputs            : ['nl_2_fmt_output_array']

As illustrated in the figure 4, the implemented c-graph can be considered as a sequential graph, managed as a simple linked list. Fixed-executing order is defined by the C-code optimizer engine according two main criteria: data-path dependencies (or IO tensor dependencies) and the minimization of the RAM memory peak usage.

Each computational c-node is entirely defined by:

  • operation type, parameters
  • input tensors list: [I]
  • optional weights/bias tensors list: [W]
  • optional scratches tensors list: [S]
  • outputs tensors list: [O]

C-Arrays table

'C-Arrays' table lists the objects allowing to handle the base address, size and meta-data of the data memory segments for the different tensors. For each item, number of item, size in byte (item/size), memory segment location (mem-pool), type (c-type) and short format description (fmt) are reported.

Validate command

Description

The 'validate' command allows to import, to render and to validate the generated C-files. Please refer to [2], “Validation engine” and “AI validation application” sections to have an overview of the underlying process. In particular for the validation on target ('--mode stm32'), the STM32 board should be flashed with a validation firmware including the model. Detailed description of the used metrics are described in [6].

Note — Be aware, that the main purpose of the underlying validation process is to test the generated C files with the associated network runtime library (desktop/X86 or STM32 run-time) by comparison with the imported model. Subsequently, only a representative and limited part of a whole validation or test data set can be used. It has not been designed to valid a pre-trained model as during a training/test phase.

Specific arguments

parameter description
--mode indicates the mode of validation. x86 (default value) performs a validation on desktop. stm32/ stm32_io_only is used to perform a validation on target. - Optional
-vi/valinput indicates the custom test data set which must be used. If not defined an internal self-generated random data set is used (refer to [6], “Input validation files” section) - Optional
-vo/valoutput indicates the expected custom output values. If the data are already provided in a simple file (*.npz) through the -vi option this argument is skipped - Optional
-b/--batches indicates how many random data sample is generated (default: 10) - Optional
-d/--desc describes the COM port which is used to communicate with a STM32 board (see “desc argument” section) - Optional
--full if defined, this flag indicates that an extended validation process is applied to report the L2r error layer-by-layer. Else only the L2r is evaluated on the last or output layers. - Optional For the x86, this allows also to report the relative executing time by layer. - Optional
--validate.batch_mode when an input custom data set is used, this argument is used to limit the number of samples. Two modes are possible: first to indicate that only the first batches samples are used. random indicates that batches samples are randomly selected with a fixed seed (see last example of the “Examples” section) - Optional
--classifier if defined, this flag indicates that the provided model should be considered as a classifier vs regressor. This implies that the computation of the 'CM' and 'ACC' metrics will be evaluated, else an auto-detection mechanism is used to evaluate if the model is a classifier or not. - Optional
--no-check if defined and combined with the 'stm32' mode, this “debug” flag allows to reduce the full preliminary check-list to make sure that the flashed STM32 C-model has been generated with the same tools and options. Only the c-name and network IO shape/format are checked. - Optional

Examples

  • Minimal command to validate a 32b float model with the self-generated random input data (“Validation on desktop”).

    $ stm32ai validate -m <model_f32p_file_path>

  • To report the “L2r error” and relative execution time by layer (“Validation on desktop”).

    $ stm32ai validate -m <model_f32p_file_path> --full

  • Minimal command to validate a 32b float model on STM32 target. Note that a complete profiling report including execution time by layer is generated by default.

    $ stm32ai validate -m <model_f32p_file_path> --mode stm32

  • Validation of a 32b float model with self-generated random input data and compression factor (“Validation on desktop”)

    $ stm32ai validate -m <model_f32p_file_path> -c 4

  • Validate a model with a custom data set

    $ stm32ai validate -m <model_file_path> -vi test_data.csv

  • Validate a quantized model with a custom data set

    $ stm32ai validate -m <modified_model_file>.h5 -q <quant_file_desc>.json -vi test_data.npz

  • Validate a model with only 20 randomly selected samples from a large custom data set

    $ stm32ai validate -m <modified_model_file>.h5 -vi test_large_data.npz --validate.batch_mode random -b 20

Serial COM port configuration

The '--desc/-d' argument should be used to indicate how to configure the serial COM driver to access the STM32 board. Previously, the STM32 board should be flashed with an aiValidation application (refer to [2], “AI validation application” section).

By default, an auto-detection mechanism is applied to discover a connected board at @115200 (default value: default:115200)

  • set the baud-rate to @961600

    $ stm32ai validate -m <model_file_path> --mode stm32 -d 921600

  • set the COM port to COM16 (Windows case ) or /dev/ttyACM0 (Linux case)

    $ stm32ai validate -m <model_file_path> --mode stm32 -d COM16
$ stm32ai validate -m <model_file_path> --mode stm32 -d /dev/ttyACM0

  • set the COM port to COM16 and the baud-rate @921600

    $ stm32ai validate -m <model_file_path> --mode stm32 -d COM16:921600

Generate command

Description

The 'generate' command is used to generate the specialized network and data C-files. They are generated in the specified output directory.

$ stm32ai generate -m <model_file_path> -o <output-directory-path>
...
-- Generating C-code
-- Generating C-code - done (elapsed time 0.900s)
Installing..
 <output-directory-path>\<name>.c
 <output-directory-path>\<name>_data.c
 <output-directory-path>\<name>.h
 <output-directory-path>\<name>_data.h

Creating report file <output-directory-path>\network_generate_report.txt
...

The '<name>.c/.h' files contain the topology of the C-model (C-struct definition of the tensors and the operators), including the embedded inference client API (refer to [5]) to use the generated model on the top of the optimized inference runtime library. '<name>_data.c/.h' files are by default a simple C-array with the data of the weight/bias tensors. However, the '--split-weights' option allows to have a C-array by tensor (refer to [5], “Split weights buffer” section) and the '--binary' option creates a simple binary file with the data of the weight/bias tensors. The '--relocatable' option allows to generate a relocatable binary model including the topology, the requested kernels and the weights in a single binary file (refer to [9]).

  • -v 2 argument should be used to report more information similar, to the “Analyze” command.

Specific arguments

parameter description
--binary if defined, this flag forces the generation of a binary file '<name>_data.bin'. The '<name>_data.c' and '<name>_data.h' files are always generated (see “Particular network data c-file” section). This binary file contains ONLY the data of the different weights/bias tensors, C-implementation of the topology (including the meta-data, scale/zero-point…) is always generated in the <name>.c/.h files. - Optional
--generate.fota if the --binary flag is passed, this additional argument with 'True' value allows to add a specific ST header which is expected for a partial Firmware Over-The-Air (FOTA) process. Name of the generated file is suffixed with the _fota extension: <name>_data_fota.bin. - Optional
--address with --binary flag, this helper option can be used to indicate the address where the weights will be located to generate a particular generated '<name>_data.c' file. - Optional
--copy-weights-at with --binary flag and --address option, this helper option can be used to indicate the destination address where the weights should be copied at initialization time thanks to a particular generated '<name>_data.c' file. - Optional

Specific arguments to generate a relocatable binary model (refer to [9] for details).

parameter description
-r/--relocatable if defined this option allows to generate a relocatable binary model. '--binary' option can be used to have a separate binary file with only the data of the weight/bias tensors. - Optional
--lib indicates the root directory to find the relocatable network runtime libraries. Typical value: '$X_CUBE_AI_DIR/Middlewares/ST/AI'. - Optional (but mandatory if the '--relocatable’ option is defined).
--series indicates the targeted STM32 series for the generation of the relocatable binary model. Possible values: stm32f4 (default), stm32f3, stm32l4, stm32f7, stm32h7, stm32l5, stm32wl, stm32mp1. - Optional
--no-c-files if defined, this flags avoids to generate the specific C-files. - Optional

Note that '--split-weights', '--address' and '--copy-weights-at' options are not supported with the '--relocatable' option.

Particular network data c-file

The helper '--address' and '--copy-weights-at' arguments are the convenience options to generate a specific ai_network_data_weights_get() function. The returned address should be passed to the ai_<network>_init() function through the ai_network_params structure (refer to [5]). Note that this (including copy function) can be fully managed by the application code.

If the --binary (or --relocatable) option is passed without the '--address' or '--copy-weights-at' arguments, following network_data.c file is generated

#include "network_data.h"

ai_handle ai_network_data_weights_get(void)
{
  return AI_HANDLE_NULL;
}

Example of generated network_data.c file with the --binary and --address 0x810000 options.

#include "network_data.h"

#define AI_NETWORK_DATA_ADDR 0x810000

ai_handle ai_network_data_weights_get(void)
{
  return AI_HANDLE_PTR(AI_NETWORK_DATA_ADDR);
}

Example of generated network_data.c file with the --binary, --address 0x810000 and --copy-weights-at 0cD0000000 options.

#include <string.h>
#include "network_data.h"

#define AI_NETWORK_DATA_ADDR 0x81000
#define AI_NETWORK_DATA_DST_ADDR 0cD0000000

ai_handle ai_network_data_weights_get(void)
{
  memcpy((void *)AI_NETWORK_DATA_DST_ADDR, (const void *)AI_NETWORK_DATA_ADDR,
                                            AI_NETWORK_DATA_WEIGHTS_SIZE);
  return AI_HANDLE_PTR(AI_NETWORK_DATA_DST_ADDR);
}

FOTA support

The '--binary' flag allows to implement a simple/limited mechanism to be able to update a model w/o having to do a full firmware update. Only the weights (raw data) for a floating-point model is supported. The topology changes or configuration modifications are not supported. Consequently, a quantize model can be not updated because the tensor definitions are included in the '<network>.c' file (part of the generated C-struct to handle the model).

The '--relocatable' flags (refer to [9]) allows to implement a more flexible support to upgrade a whole C-model, including the used forward kernel functions w/o having to do a full firmware update.

Update an ioc-based project

For a X-CUBE-AI IDE project (ioc-based), the user has the possibility to update only the generated NN C-files. In this case, the '--output' option is used to indicate the root directory of the IDE project, i.e. location of the '.ioc' file. The destination of the previous NN c-files are automatically discovered in the source tree else output directory is used.

The '--output' option is used to indicate the single destination of the generated NN C-files. An empty file with the '.ioc' extension can be defined in the root directory of the custom source tree to use the discovery mechanism as for the update of an ioc-based project.
