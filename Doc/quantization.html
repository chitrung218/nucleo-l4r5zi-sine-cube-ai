Introduction

Quantization support

X-CUBE-AI code generator can be used to generate and to deploy a quantized model (8b integer format). Quantization (also called calibration) is an optimization technique to deploy a 32b float model reducing the size (smaller storage size and less memory peak usage at runtime) and improving CPU/MCU and hardware accelerator latency (including power consumption aspects) with little degradation in model accuracy. It is an important part of various optimization techniques: topology-oriented, features-map reduction, pruning, weights compression… which can be applied to address the resource-constrained environment.

There are two forms of quantization: post-training quantization and quantization aware training. First is easier to use, it allows to quantize a pre-trained model with a limited and representative data set. Quantization aware training is done during the training process and is often better for model accuracy.

The CLI provides an internal post-training quantization process (see “quantize” command section) with different quantization schemes for an already-trained Keras model.

X-CUBE-AI is able to import two sort of quantized model:

  • a Keras float model associated with its tensor format configuration file. The conversion of each 32b float weight/bias tensors to 8b integer format is directly achieved by the importer engine thanks to the provided settings. No calibration is performed.
  • a quantized TensorFlow lite model (generated by a post-training or training aware process). In this case, the conversion has been performed by the TensorFlow Lite framework, principally through the “TFLite converter” utility exporting the TensorFlow lite file. No calibration is performed.

For a given layer/operator, weights and activations should be quantized. Full 8b integer format is requested, weights only or float16 TFLite quantization variants are not supported. The mixed models with the convert operators (like QUANTIZE or DEQUANTIZE Tensor Lite operators), explicitly defined or automatically inserted by the X-CUBE-AI code generator are supported. Finally the quantized tensors are mapped on the optimized and specialized C-implementation for the supported operators (see [3]), otherwise the floating-point version of the operator is used.

Note — As the Keras API is now fully integrated in the TensorFlow framework, the user has the possibility to quantize a Keras model with the TFLite Converter utility (post-training quantization) and/or the X-CUBE-AI internal process. Despite the current limitations, this internal process offers more quantization schemes which can be more interesting in term of execution time and precision (i.e. accuracy). Results are fully dependent of the model size and associated topology. It provides also currently a better support to deploy a model with the recurrent layers which are kept in float w/o extra manipulation.

The “analyze”, “validate” and “generate” commands can be used w/o limitations. The '-q/--quantize' argument is used to pass the specific “tensor format configuration” file for a given Keras model.

$ stm32ai analyze -m <reshaped_model_file>.h5 -q <quant_file_desc>.json
$ stm32ai analyze -m <quantized_model_file>.tflite
$ stm32ai validate -m <quantized_model>.tflite -vi test_data.npz

Note — By default, if a Softmax layer is part of the network, it is automatically keep in float.

Supported 8b integer format

X-CUBE-AI supports two 8b integer-base arithmetics: Qm,n and Integer.

Legacy Qm,n arithmetic (introduced in X-CUBE-AI 4.0) is a signed fixed-point number format (two’s complement) where the number of fractional bits n and the number of integer bits m are specified with a constant and fixed resolution. For example, Q0.7 number has 7 fractional bits; Q1.6 number has 1 integer bit and 6 fractional bits. By convention m does not include the sign bit.

Note – Qm,n arithmetic is a legacy support, it will be deprecated in future release.

Integer arithmetic (introduced in X-CUBE-AI 4.1) is based on a more representative convention used by Google for the quantized models. See the following reference to highlight the underlying rational.

  • Quantization and Training of Neural Networks for Efficient Integer-Arithmetic-Only Inference (https://arxiv.org/abs/1712.05877)

For the integer arithmetic, each real number r is represented in function of the quantized value q, a scale factor (arbitrary positive real number) and a zero_point parameter. Quantization scheme is an affine mapping of the integers q to real numbers r.

zero_point has the same integer C-type like the q data.

For Qm,n, fixed point format with power of 2 scaling, centered to zero, the precision is directly deduced and fixed by the m,n parameters. For the Integer arithmetic, precision is dependent of a scale factor and the quantized values are linearly distributed around the zero_point value. In both case, resolution/precision is constant vs floating-point representation.

Per-axis (or per-channel) vs per-tensor

Currently per-tensor quantization is supported for Qm,n and Integer arithmetics. This means that the same format is used for the entire tensor. For Integer arithemtic only, per-axis (or per-channel) for conv-base operator is supported, this means there will be one scale and/or zero_point per slice.

Activation tensors are always in per-tensor.

Symmetric vs Asymmetric

Asymmetric means that the tensor can have zero_point anywhere within the signed 8b range [-128, 127] or unsigned 8b range [0, 255]. Symmetric means that the tensor is forced to have zero_point equal to zero. By enforcing zero_point to zero, some kernel optimization implementations are possible to limit the cost of the operations (off-line pre-calculation,…). By nature, the activations are asymmetric, consequently symmetric format for the activations is not supported. For the weights/bias, asymmetric and symmetric format are supported.

For Qm,n arithmetic, only symmetric format is considered.

Signed integer vs Unsigned integer - supported schemes

Signed or unsigned integer type can be defined for the weights and/or activations. However all requested kernels are not implemented or relevant to support the different optimized combinations related to the symmetric and asymmetric format. This imply that only the following integer schemes or combinations are supported:

scheme weights activations
ua/ua unsigned and asymmetric unsigned and asymmetric
ss/sa signed and symmetric signed and asymmetric
ss/ua signed and symmetric unsigned and asymmetric

For Qm,n arithmetic, only equivalent “ss/ss” scheme is considered. Implementation of the kernels are specific.

TensorFlow lite support

X-CUBE-AI is able to import the training-aware and post-training quantized TensorFlow lite models. Post-training quantized models (TensorFlow v1.15 or v2.x) are based on the “ss/sa” and per-channel scheme. Activations are asymmetric and signed (int8), weights/bias are symmetric and signed (int8). Previous training-aware quantized models are based on the “ua/ua” scheme. Now the “ss/sa” and per-channel scheme is also the privileged scheme to address efficiently the Coral Edge TPUs or TensorFlow Lite for Microcontrollers runtime.

Note — TensorFlow Lite framework is used to deploy a deep learning model on mobile and embedded devices. Generated TFLite file is a self-contained file containing a frozen description of the graph, setting of the operators and the tensors (including the data). 32b float and quantized models are supported. This file is directly used by a runtime interpreter, see TensorFlow Lite for Microcontrollers, or as entry point for a compiler like Coral Edge TPUs compiler or code generator like X-CUBE-AI to create an adapted and optimized version targeting a particular hardware: MPU, MCU or hardware assist IP. Functional point of view, content of the file is similar to the generate <network>.c and <network>_data.c files, implementation of the kernels should be provided by the runtime.

Quantize command

Overview

The “quantize” command allows to perform a post-training quantization process on a 32b float Keras model allowing to generate a new Keras model (reshaped model) file and associated tensor format configuration file compatible with the X-CUBE-AI code generator. Options are passed through a post-training quantization configuration json file ('-q/--quantize' argument). An additional JSON file is generated because Keras h5 format does not provide natively a support to handle quantized params or meta information. Note that the reshaped model file is basically an un-fused version of the original 32b float model which can be used as-is.

Post-training quantization process limitations

  • residual or multi-branches model
  • model with multiple inputs or outputs
  • only channel last (NHWC) tensor representation is supported
  • only the supported floating-point Keras operators (refer to [4]) can be quantized. Layers that are not supported for fixed-point conversion will be kept in floating point.

Examples

  • Perform the post-training quantization process on an already-trained floating-point Keras model.

    $ stm32ai quantize -q <conf_quant>.json

  • Validate a Keras model after Keras post-training quantization

    $ stm32ai validate -m <reshaped_model_file>.h5 -q <quant_file_desc>.json -vi test_data.npz

  • Generate the specialized files for a quantized Keras model.

    $ stm32ai generate -m <expanded_model_file>.h5 -q <quant_file_desc>.json

The use the Keras post-training quantization process a configuration file (JSON dictionary) is requested with the following keys:

“output_directory” indicates the root directory to store the results. Produced files are stored in the following directory: <output_directory>/<model_name>_<algorithm>_<date>_<time>/.
“modules_directory” indicates the directory containing the user test-set generation and optional user quantizer Python files.
“filename_test_set_generation” name of the file (with or without py extension), where the user writes a potential pre-processing of the data, and in any case, load the test-sets into generators. This file is mandatory.
“filename_quantizer_algos” name of the file (with or without py extension), where the user can write his own quantizer. This is only needed if “User” algorithm is requested.

Note – For the “path_to_floatingpoint_h5”, “output_directory”, “modules_directory”, “quant_test_set_dir” and “evaluation_test_set_dir” keys if the values are not prefixed by "./" or "/", path is relative to the location of the JSON file path. Otherwise, the path is absolute or relative to the current executing path where the stm32ai application is launched.

Note – All fields are requested. Only the “filename_quantizer_algos” can be omitted if “User” algo is not selected.

“quant_test_ratio” parameter

If you have 1000 images in “quant_test_set_dir” and the user sets “quant_test_ratio” to 0.8 then only 800 images we be used from the quantization test-set. This parameter can be viewed as a way to control the execution time of the script that can be long depending on the user system, the size of the quantization test-set, the quantization algorithm and the depth of the network. Please, note that in case Keras pre-existing ImageDataGenerator() class is used to load data, then Keras constraints impose “quant_test_ratio” be strictly lower than 1.

Supported Keras layers

Only the following layers can be quantized by the quantization process:

  • Trainable Layers:
    • Dense, Conv2D, DepthwiseConv2D, DepthwiseConv1D, SeparableConv2D (will be split into DepthwiseConv2D + pointwise Conv2D), Conv1D with the following exceptions:
      • “dilation”: values different from 1 are not supported
  • Pooling layers:
    • MaxPooling1D, MaxPooling2D, AveragePooling1D, AveragePooling2D, GlobalMaxPooling1D, GlobalMaxPooling2D, GlobalAveragePooling1D, GlobalAveragePooling2D
  • Activations:
    • Linear, ReLU, Sigmoid, Tanh
    • Advanced: LeakyReLU, PReLU, ELU, ThresholdedReLU
  • Various
    • Input Layer

Permute, Flatten, Reshape, RepeatVector layers are not quantized: they are just re-formatting their input tensor. Dropout and SpatialDropout are skipped. Layers that are not supported for fixed-point conversion will be kept in floating point.

  • Only channel-last (NHWC) tensor representation is supported.

Test-set considerations

The quantization process gives the possibility to load batches of test vectors. The quantization requires the knowledge of the range of the values of all the tensors. The weights are constant and thus their range is easy to estimate, whereas the activation’s range depends on the input data. An accurate estimate of the activation range requires a test set representative of real data:

  • it means a balanced test set over all the labels to be classified
  • test set should be large enough to reflect how variable can be the network inputs and thus to be as representative as possible

We call it quantization test-set. If this test-set is too small, not representative enough or even unbalanced, quantization will be performed as usual but there is a risk that the quantized model performance be not satisfying. The other test-set, called evaluation test-set, is used to assess performance of floating point model and quantized model and potentially compare them. The evaluation test-set and the quantization test-set should be as independent as possible to avoid over-estimation of the quantized model performance.

The input vectors is loaded with the create_test_generator() function which is dynamically imported by the core from the user “filename_test_set_generation.py” module. Preprocessing or adaptation of the loaded data should be done in this function (see “Quantize a NMIST model” sections).

def create_test_generator(quant_test_set_dir, evaluation_test_set_dir,
                          quant_test_ratio, batch_size)

Note — If the neural network is not used for a classification task, or if labels are not available for the quantizer then in the create_test_generator() function set quant_test_set_labeled to None and eval_test_set_labeled to None.

Quantizers

Two quantizers (also called algorithms) are provided to generate the tensor format configuration file: “Minmax” and “Greedy”. The user (“User” algorithm) has also the possibility to provide its own quantization method with different compromises between quantized data precision and saturation, by implementing the estimate method in the quantizer class and passing it to the functions for weight and activation quantization (see “filename_quantizer_algos.py” user module).

  • “Minmax” invokes the simple and quick quantization process based on min and max of all the tensors for weights and activations. The quantization test-set is used to estimate the activation ranges. The weights are constant and thus their ranges are directly estimated.

  • “Greedy” invokes an iterative process which tries to improve the picture for activations only, by searching on a per layer basis among a set of possible quantization formats which is the best in terms of classification accuracy. Note that only the accuracy is considered and not the loss function.

    • consequently, if there is no label in the quantization test-set then “Greedy” will not be executed.
    • be careful, the execution of this algorithm takes much more time than the simple “Minmax” but can also lead to better performance.

Note — “Greedy” is not available for the “Integer” arithmetic, just “User” or “Minmax”.

Note — Whatever the algorithm used, the quantizer script issues the quantized model expected accuracy on the evaluation test-set (if labels are present). This is an interesting indication, that may be used to compare different quantizers but the recommendation is to verify that the quantized model generalize well-enough according to your requirements. This has to be done with real field inputs.

Tensor format configuration file

The proprietary tensor format configuration file is a JSON dictionary giving the expected tensor format. One entry is defined for each quantized tensor. If a tensor is omitted, the format is float by default (unless it is inferred). As a result, there is no way to indicate that a tensor format is float. The configuration is provided as a JSON file generated from the network structure and it is specific to a neural network model.

{
    "version": "2.0",
    "<layer_type>_<idx>_<tensor_name>": {
        "format": {
            "class": "Integer",
            "type": "S8",
            "params": [ [ 0.0019106452371559892 ],[ 132 ] ],
        }
    },
}
field description
version version/format of the JSON file
name <layer_type>: name/type of tf.keras layer
<idx>: position of the layer in the Keras network
<tensor_name>: designates the conventional name of the associated tensor: “out”, “weights”, “bias”.
format.class indicates the arithmetic format: can be “FXP” (for “Qmn”) or “Integer”
format.type indicates the type of data: “U8”, “S8” or “S32” (bias in integer)
format.params indicates the parameters “FXP”: [number of integer bits (M), number of fractional bits (N)]
“Integer”: [[scale value], [zero_point value]]

def load_mnist(test_dir):
    import numpy as np
    import sys
    import os

    MNIST_SHAPE = (28, 28, 1)
    N_CLASSES = 10

    print('Keras version  :', keras.__version__)
    print('Python version :', sys.version)

    fdata_set = os.path.join(test_dir,'mnist.npz')
    if not os.path.isfile(fdata_set):
      print('Upload the nmist data set with Keras service...')
      mnist = keras.datasets.mnist
      _, (x_test, y_test) = mnist.load_data()
    else:
      print('Use the data set {}'.format(fdata_set))
      arrays = np.load(os.path.join(fdata_set))
      x_test, y_test = arrays['x_test'], arrays['y_test']

    msize = min(1000, len(x_test))
    np.random.seed(2)  # deterministic results
    rchoice = np.random.choice(len(x_test), size=msize, replace=False)

    x_test, y_test = x_test[rchoice], y_test[rchoice]

    x_test = x_test.reshape((-1, ) + MNIST_SHAPE).astype('float32') / 255
    y_test = to_categorical(y_test, N_CLASSES)

    print('x_test', x_test.shape)
    print('y_test', y_test.shape)

    return x_test, y_test
